  перегружает при изменении. `-Once` — разовый.
- **Пути — в `config.txt`** рядом со скриптом (создаётся сам, если нет).

- **Веб-режим** (`-Web`): локальный HTTP-сервер + браузер с тем же отчётом в виде HTML-дашборда
  (авто-обновление). Только встроенный .NET, без админ-прав и сторонних пакетов.

## 3. Структура файла плана (`daily shift dpp`)

- Строка 1: `Programmed Production for VEURNE (cases)`, метки `WEEK`/`start`/`stop` + значения
  (start/stop = серийные даты Excel; неделя Sun→Sun, 8 дней).
- Строка 2: заголовки. Колонка **B = код материала (SKU, 3400xxxxx)**, **C = описание**.
  Далее **датированные колонки**: по **3 смены на день** (заголовок строки 2 = серийная дата дня).
- **3 датированные колонки на день = задания смен 1 / 2 / 3** (по порядку; проверено:
  Mon27 план 1157/1234/1234 ≈ факт 1039/1217/1332). Старт дня 05:00 — как и в boxprinting.
- Столбцы после датированных (`AB`, `AC` ≈ col 28/29) = **недельный итог** (дублируется),
  `AD` = ABC-сегмент `---`.
- Итог недели по SKU = сумма датированных колонок (= AB/AC).

## 4. Структура `L9RCDB` (и других `*RCDB`) в boxprinting — КЛЮЧЕВОЕ

- Лист = «L9 datatabel voor **31 dagen**». Размер ~293×165.
- Организован **блоками по 5 колонок**: `[uur, SAP, doosprint, Machine, aantal]`.
  Блоки начинаются с колонок 1,6,11,… (в 0-based pyxlsb: c%5==0; в 1-based COM: (c-1)%5==0).
- **Строка 2** над каждым блоком = **номер дня месяца (1..31)**.
- В **строках данных (с 4-й)** колонка-начало блока = **uur (час 0..23)**;
  `SAP` (start+1) = код материала; `aantal` (start+4) = число коробок;
  `doosprint` = ZPL-строка этикетки (содержит `14|3400xxxxx` = материал, `23|NNN` = счётчик).
- **Производственный день стартует в 05:00**: блок дня D содержит часы 5..23 дня D + часы 0..4 дня D+1.
  Проверено: блок дня 27 имеет часы 0–23, блок дня 28 (снимок обрезан) — только 5–22.
- Смены целиком внутри одного блока дня: 1 = uur 5–12, 2 = 13–20, 3 = 21,22,23,0,1,2,3,4.
- «Мусорные» первые блоки (служебная зона) сами отфильтровываются: там в колонке SAP не код
  материала, поэтому совпадений с SKU нет.
- Сентинелы: `uur = 25` и `-` = пустые/заглушки (в строках с SAP=SKU не встречаются).

**Факт по SKU** = сумма `aantal`, где `SAP == SKU`, сгруппированная по дню месяца (и при желании по часу→смене).

## 5. Сопоставление план ↔ факт

- По **дню месяца**: дата дня плана → `.Day` → блок с тем же номером в `L9RCDB`.
- Внутри ~31-дневного буфера день месяца уникален; план и факт должны быть одного месяца (для свежих недель так и есть). Это ограничение, не баг — так устроен сам сбор (boxprinting перезаписывает блок по дню месяца).

### Проверенные факты (валидация, нед.18, SKU 340062773, L9RCDB)
| День | План | Факт |
|---|---|---|
| 27 | 3625 | 3588 |
| 28 | 2622 | 2675 |
| неделя | 6247 | 6264 (100,3%) |

Смены дня 27: 1=1039, 2=1217, 3=1332 (сумма 3588 ✓).
Файл-снимок сохранён **30/04/2026 19:13**, поэтому день 30 имеет часы 5–19; на 30-е линия L9 шла
`340064833` (1141) и `340018283` (571), а не 340062773.

## 6. Как работает скрипт `Read-WeekPlan.ps1`

- Читает Excel через **COM** (`New-Object -ComObject Excel.Application`, ReadOnly), `Range(...).Value2`,
  доступ `$arr.GetValue($r,$c)` (НЕ `$arr[$r,$c]` — PowerShell неоднозначно парсит).
- **Пути к данным — из `config.txt`** (рядом со скриптом): ключи `BoxPrintingFile`, `PlanFolder`,
  `RcdbSheet`. Нет файла → `New-ConfigFile` создаёт его с дефолтами (UTF-8 BOM). `Read-ConfigFile`
  парсит `key = value` (# — комментарии). Приоритет: **параметр > config.txt > дефолт**
  (`$script:HasBox/HasRcdb`, `$script:PlanFolder`). Файл плана: авто-поиск `daily shift NDwk*.xls*`
  в `PlanFolder`, номер недели `NDwk0*(\d+)`; без `-Week` — максимальная неделя.
- `Get-Production` → `$prod[день][час][sku] = aantal` (по ВСЕМ SKU). `Detect-CurrentSku`/`Shift-Made`/
  `Day-Made` агрегируют. `Get-PlanColumns` → список {Col,Date,Shift0..2}; `Get-PlanRowForSku`,
  `Get-SkuFromPlan` (SKU с макс. планом на дату+смену).
- **Мульти-SKU (устойчивость к переналадке):** `Get-ShiftSkus` = все SKU с продукцией в текущей
  смене (до сейчас) → таблица «SKU: gemaakt/plan/nog te doen» по каждому. `Detect-CurrentSku` =
  доминирующий SKU **последнего часа** (порядок `HourOrder`: 5..23→0..18, 0..4→19..23), помечается
  `<< nu` и получает недельную сводку. Фолбэк (ничего не сделано) — `Get-SkuFromPlan`, НО **только
  среди SKU, реально встречающихся в истории L9RCDB** (`Get-LineSkus $prod`), т.к. план общезаводской
  и без фильтра по линии подставлял продукт чужой линии (баг с «DORITOS на L9»). Нет истории/нет
  файла → фолбэк = `$null` → «Geen SKU». `-Sku` = фокус на одном коде. `Get-PlanShift` = план SKU.
- **Отсечка «по сейчас»:** `$nowOrder = HourOrder($hNow)`; все «сегодняшние» суммы и детект берут
  только часы с `HourOrder <= nowOrder`; в week-made будущие даты пропускаются. Для живого режима
  это no-op (данных из будущего нет); для `-Now`-симуляции делает картину корректной «на момент».
- **Названия смен X/Y/Z** (`$d.ShiftCode`, в `$d.ShiftLabel = "ploeg <code> (<time>)"`): nacht(shift3)=Z;
  namiddag(shift2)= Y если `$d.WeekNo % 2 -eq 0` иначе X; ochtend(shift1)= обратное. Считается ПОСЛЕ
  чтения weekNo из плана (shiftNo определён по часу раньше). weekNo нет → `$even=false` (namiddag=X).
- **Прогноз конца смены** (`$d.HasForecast`, `Fc*`): для текущего SKU, темп **с момента запуска SKU**
  (`firstHour = Get-SkuFirstHour`, `skuStart = HourToDateTime`). `elapsed = fileTime - skuStart`;
  `tempo = curMade/elapsed`; `perHour = tempo*60`; `remaining = max(0, shiftEnd - fileTime)`
  (shiftEnd = ploegstart+8ч); **`projection = curMade + tempo*remaining`** (окно SKU до конца смены;
  <8ч при запуске в середине; для SKU всю смену = старому `tempo*480`). **«Сейчас» = LastWriteTime
  файла boxprinting** (`$fileTime`), НЕ Get-Date. ploegstart = `$prodDate` + (5/13/21)ч. Прогноза нет,
  если файла нет / elapsed<1 / curMade=0. Вывод: блок «PROGNOSE einde ploeg» + карточки (web).
- **Таблица смены:** по каждому SKU из `Get-ShiftSkus` строка `gemaakt / plan / nog te doen` + %:
  `gemaakt` = `Shift-Made` (до сейчас), `plan` = `Get-PlanShift` (план этого SKU на дату+смену),
  `nog te doen` = plan − gemaakt. При >1 SKU добавляется строка `Totaal`. Недельный блок строит
  `planByDate` (date→@(s1,s2,s3)) для текущего SKU и считает план/факт с отсечкой по «сейчас».
- Нормализация SAP: `Format-Sap` (trim + срез `.0`); `Is-Sku` = 9 цифр, начинается `3400`.
- «Сейчас»: `Get-Date` (или `-Now`); час → смена; если час<5 → производственный день = вчера.
- **Режим монитора (по умолчанию):** бесконечный цикл, раз в `$IntervalSeconds` сверяет
  `LastWriteTime` boxprinting; при изменении — `Clear-Host` + `Invoke-Report`. `-Once` = разовый.
  Excel создаётся/закрывается ВНУТРИ каждого `Invoke-Report` (в ожидании не висит → нет
  осиротевших `EXCEL.EXE` при Ctrl+C).
- **Разделение вычисление/рендер:** `Get-ReportData` (открывает Excel, считает, возвращает
  объект; ошибки в `.Error`, предупреждения в `.Warning`) → `Render-Console` (Write-Host) и
  `Render-Html` (HTML-строка). `Invoke-Report` = `Render-Console (Get-ReportData)`.
- **Веб-сервер** (`Start-WebServer`, при `-Web`): `System.Net.Sockets.TcpListener` на
  `127.0.0.1:$Port` (без админ-прав/urlacl, в отличие от HttpListener), `Start-Process $url`
  открывает браузер (`-NoBrowser` отключает). На запрос: читает запрос (best-effort), отдаёт
  HTML с заголовками HTTP вручную. **Кэш**: пересчёт только если изменился `LastWriteTime`
  boxprinting ИЛИ прошло >15с (чтобы не открывать Excel на каждый favicon/refresh). В странице
  `<meta http-equiv='refresh' content='$IntervalSeconds'>`. `HtmlEnc` экранирует текст.
- Числа/даты — культура **nl-BE** (`6.247`, `100,3`, дни `ma/di/wo`).

## 7. Подводные камни

- **`.ps1` обязательно UTF-8 с BOM**, иначе Windows PowerShell 5.1 ломает кириллицу/нидерландский.
  После каждого редактирования пересохранять с BOM:
  `[IO.File]::WriteAllText($p,[IO.File]::ReadAllText($p,[Text.Encoding]::UTF8),(New-Object Text.UTF8Encoding($true)))`
- `$PSBoundParameters` внутри функции ≠ скриптовый → флаги переданных параметров зафиксированы
  на скриптовом уровне: `$script:HasNow/HasWeek/HasPlanFile/HasSku/HasBox/HasRcdb`.
- Config-функции (`Read-ConfigFile`/`New-ConfigFile`) определены ДО блока загрузки конфига; сам
  блок resolve стоит после всех функций, перед секцией запуска (значения нужны в `Invoke-Report`).
- 2D-массив от `.Value2` индексируется `$arr.GetValue($r,$c)`, НЕ `$arr[$r,$c]`.
- `Start-Process` со ссылками с пробелами: аргументы передавать **одной строкой** с кавычками
  вокруг путей (иначе обрезается на пробеле).
- `Clear-Host` при перенаправленном выводе кидает исключение → обёрнут в try/catch.
- Линия задаётся `-RcdbSheet` (по умолчанию L9RCDB). Неверная линия → факт 0.

## 8b. Второй инструмент: `Test-BoxCount.ps1` (лист `Boxruw9`, коробки поштучно)

Отдельный скрипт (запуск: `Test-BoxCount-Web.cmd` → порт 8771; `Test-BoxCount-Web-2010.cmd` →
порт 8772 с `-Now "2026-07-21 20:10"`). План недели НЕ читает.

- **Источник:** лист `Boxruw9` (не `*RCDB`!) — сырая дельта-выгрузка ActiveFactory.
  Данные с **строки 11**: `A` = время (Excel-serial), `B` = ZPL-этикетка, `C` = `DELTA`.
  Читается `A11:C<lastRow>` одним `Value2`, парсинг до **первой пустой B**. Одна строка = коробка.
  Тип продукта = тег `14`; счётчик коробки = последнее поле после последнего `|`.
- **ЛОВУШКА (проверено на снимках 13/07 и 21/07):** **первая строка данных — не коробка**, а
  «значение на начало окна» дельта-выгрузки: время ровно `05:00:00.000` и часто ЧУЖОЙ продукт
  предыдущей смены (13/07: `340062773` в 05:00, реальный старт `340056955` в 05:55).
  Скрипт её пропускает (проверка `$i -eq 1` + минуты/секунды/мс = 0) → `StartRowSkipped`.
- Строгие правила чтения (важно, файл — рабочий кеш завода): ReadOnly + assert, `EnableEvents=$false`
  (иначе `Workbook_Open` включает `xlAutomatic` и файл пересчитывается), `AutomationSecurity=ForceDisable`,
  `Calculation=xlManual` через пустую 1-ю книгу, в конце — `Quit` + добивание СВОЕГО PID (HWND→PID),
  чужой Excel пользователя не трогаем.
- **Прогноз конца смены** (перенесён из `Read-WeekPlan.ps1`, но точнее — на уровне коробок):
  - **Основной продукт = продукт ПОСЛЕДНЕЙ строки** (`$sProd[last]`), НЕ максимум по количеству
    (явное требование пользователя). Помечается `<< nu` / бейджем `nu`.
  - Темп берётся с **начала текущего непрерывного run** этого продукта: идём назад по строкам,
    пока продукт тот же (`$idx`), `runStart = $sTs[$idx]`, `perMin = runCount / (refNow - runStart)`.
  - `refNow = min(now, LastWriteTime файла)`, обрезанный ploegeinde и не раньше последней коробки.
  - `ProjTotal = Total + perMin * remain`, `ProjMain = MainCount + perMin * remain`;
    вторая оценка `ProjRecent` по темпу последних `-RecentMinutes` (деф. 30) минут по всей линии;
    `NeedPerMin` = сколько нужно для target; `EtaText` = когда добьём target (+ «na ploegeinde»).
  - `FcWeak` (run < 5 мин или < 5 коробок) → пометка «voorlopig».
- **`-Now` = машина времени:** строки позже указанного момента не считаются (`$cutoff`), поэтому
  на снимке можно честно смотреть «как было в 15:00».
- **Target — из недельного плана** (`Read-PlanTargets`, добавлено 2026-07-22): открывает
  `daily shift NDwk*.xls*` (лист `daily shift dpp`) ТЕМ ЖЕ экземпляром Excel, read-only; из строки 1
  берёт `WEEK/start/stop`, из строки 2 — датированные колонки (3 на день = смены 1/2/3), находит
  колонку `дата productiedag + (shiftNo-1)`. **Target = сумма плана ТОЛЬКО тех SKU, что реально шли
  в смене** (план общезаводской: 187 строк по всем линиям — без фильтра сумма бессмысленна);
  дубли одной SAP-строки складываются. Пусто в смене → SKU с максимальным планом на день+смену
  среди `$allProds` (продукты из этого же Boxruw-листа) = «что должно идти». Дня нет в файле →
  `Covered=$false` + предупреждение + откат на config. Приоритет: `-ShiftTarget` > план > config > 1234.
  `-PlanFile`/`PlanFolder`/`-NoPlan`. Файлы-кандидаты: до 3 штук, сортировка по номеру недели из имени
  (`NDwk0*(\d+)`), берётся первый, покрывающий дату. Источник target пишется в `$d.TargetSource`.

  21/07 (неделя 30) → корректный откат на 1234 с предупреждением. Строки таблицы получили

  + зелёная линия факт-темпа + затенение будущего; `New-ForecastChartSvg` — накопительная кривая
  до «сейчас», пунктир прогноза до конца смены, наклонная target-линия, тонкий пунктир — прогноз
  по recent-темпу; `New-StopStripSvg` — полоса «идёт/стоит» на всю смену; `New-BehindChartSvg` —
  отставание/опережение относительно ровного темпа target (ноль = точно по заданию).
- **Простои (`-StopMinutes`, деф. 2 мин):** разрыв между соседними коробками > порога = стоп;
  отдельно считаются `opstart` (начало смены → первая коробка) и `nu` (последняя коробка → refNow,
  т.е. линия стоит прямо сейчас). Отсюда `StopMin/StopCount/LongestMin`, `RunMin` (= elapsed − простои),
  `AvailPct`, **`NetPerMin` = коробки / RunMin** (нетто-темп «на ходу» — совпадает с медианным тактом:
  21,0 с ⇒ 2,87/мин) и `LostBoxes = StopMin × NetPerMin`. `BehindNow = Total − TargetPerMin × elapsed`,
  `BehindEnd = ProjTotal − Target`. Валидация на 21/07 13-21: 62 мин в 8 стопах (39,5 мин в 17:02-17:42),
  Draaitijd 85,3 %, нетто 2,87/мин, потеря ≈178 коробок → это и есть разрыв до target (прогноз 1177).
  Что ещё можно построить из этих же данных (обсуждалось, НЕ сделано): коробки по часам,
  гистограмма такта, скользящий темп 15 мин, конус прогноза, тепловая карта час×минута, все 3 смены
  за сутки, стек по продуктам при переналадке. Нельзя: разрез по машинам (тег 8 всегда `901`).
- **ЛОВУШКА PowerShell:** переменные регистро-НЕзависимы — `$x = & $X $m` затирает scriptblock `$X`
  («The term '52' is not recognized»). В `New-ForecastChartSvg` помощники названы `$MapX`/`$MapY`.
- Проверено на снимках в папке: 21/07 20:10 → 1.035 коробок, темп 2,47/мин, прогноз 1.177 (95,3% от
  1.234); 13/07 10:30 → 602 коробки (фантомная строка 05:00 отброшена), прогноз 931. Кросс-проверка
  счётчиков сделана независимо через `pyxlsb`.
